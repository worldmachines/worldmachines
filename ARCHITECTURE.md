# Website Architecture

This document describes the full technical stack for worldmachines.org — how it is built, deployed, and operated. Read this before making any infrastructure changes, and keep it up to date when things change.

---

## Stack overview

| Layer | Technology | Purpose |
|-------|-----------|---------|
| Hosting | Cloudflare Pages | Static file serving + serverless Functions |
| Auth | Cloudflare Access (Zero Trust) | Email OTP gate for `/submit`, `/profile`, `/admin/*` |
| Contributor registry | Cloudflare KV (`HANDLES`) | Maps contributor emails → handles + profile data |
| Notes corpus | Cloudflare R2 (`NOTES`) | Stores `notes.parquet` for Oracle retrieval |
| Oracle AI | Cloudflare Workers AI | Embedding (`EmbeddingGemma-300M`) + chat (`Gemma-4-26b-a4b-it`) |
| Static build | Python (`scripts/build.py`) | Generates HTML pages from Markdown + article JSON |
| Article ingest | Python (`scripts/ingest.py`) + `trafilatura` | Full-text extraction, article JSON creation |
| CI/CD | GitHub Actions | Article ingestion, site rebuild, notes ingest |

The website has **no frontend build framework** — HTML, CSS, and JS are hand-authored. `build.py` is the only build step, and it only generates a few HTML files from Markdown and article JSON.

## `wrangler.jsonc` — the binding source of truth

**`website/wrangler.jsonc` is the single source of truth for all Cloudflare resource bindings.**

This is a direct-upload Pages project — it is **not** connected to GitHub for automatic deploys. Cloudflare defers entirely to `wrangler.jsonc` for binding configuration. This means:

- Anything not declared in `wrangler.jsonc` will be absent from `env` in Pages Functions at runtime.
- The "Bindings" section in the Pages dashboard will show "managed by wrangler.toml" — do not add bindings there; they will be ignored.
- When adding any new Cloudflare resource (KV namespace, R2 bucket, D1 database, etc.), always add it to `wrangler.jsonc` first.

A missing binding causes Pages Functions to throw an unhandled exception (Cloudflare error 1101) — the error is not surfaced to the user, so it will look like a silent failure or an infinite loading state.

### Current bindings

| `env` variable | Type | Resource | Used by |
|----------------|------|----------|---------|
| `HANDLES` | KV namespace | `handles` (ID: `ca021379904746528348029c242faaff`) | All contributor-aware endpoints |
| `NOTES` | R2 bucket | `worldmachines-notes` | `/api/notes-parquet` |
| `AI` | Workers AI | — | `/api/embed`, `/api/chat` |

## Deployment

### Automatic (GitHub Actions)

Two workflows deploy to Pages automatically:

**`rebuild.yml`** — triggers on push to `main` when `website/blurbs.md` or `website/devlog.md` changes:
1. Runs `scripts/build.py` to regenerate HTML.
2. Commits the rebuilt HTML files back to `main`.
3. Deploys to Pages via `wrangler pages deploy --branch main`.

**`ingest.yml`** — triggers on `repository_dispatch` event of type `article-submission` (fired by `/api/submit` when a contributor submits a link):
1. Runs `scripts/ingest.py` to extract article text and write a JSON file to `website/content/articles/`.
2. Runs `scripts/build.py` to regenerate HTML.
3. Commits changes and deploys to Pages.

### Manual

For all other changes (new pages, new Functions, `wrangler.jsonc` changes, config updates), deploy manually from the `website/` directory:

```bash
cd website
wrangler pages deploy . --project-name worldmachines --branch main
```

The `--branch main` flag is required to update the production deployment. Omitting it creates a preview deployment that does not update worldmachines.org.

### When to deploy manually

- Any change to `functions/api/`
- Any change to `wrangler.jsonc`
- New or modified HTML pages not generated by `build.py`
- Any structural site change not covered by the two auto-deploy triggers above

## Authentication

Cloudflare Access gates three areas:

| Path pattern | Who can access |
|-------------|---------------|
| `/submit` | All registered contributors |
| `/profile` | All registered contributors |
| `/admin/*` | Venkat only (admin policy) |

Access uses **email OTP** — visitors enter their email, receive a one-time code, and are admitted if their email matches an Access policy rule. On admission, Access injects a `Cf-Access-Authenticated-User-Email` header into every request, which Pages Functions read to identify the caller. There is also a JWT fallback for programmatic access.

The contributors page (`/contributors`) and all article/resource pages are **public** — no authentication required.

## Adding a contributor

Two steps are both required. Doing only one leaves the contributor either unable to log in (step 1 missing) or unable to submit (step 2 missing).

**Step 1 — Cloudflare Access (controls who can log in):**
1. dash.cloudflare.com → Zero Trust → Access → Applications
2. Find the worldmachines application protecting `/submit`
3. Edit the policy → add the contributor's email → save

**Step 2 — KV handle registry (maps their email to a handle/profile):**

Either use the admin UI at `worldmachines.org/admin/handles` (Access-gated), or write directly via wrangler:

```bash
wrangler kv key put "email@example.com" \
  '{"handle":"theirhandle","name":"Their Name","url":null,"bio":null}' \
  --namespace-id ca021379904746528348029c242faaff \
  --remote
```

## API Functions (`functions/api/`)

All Functions live in `website/functions/api/` and are served as Cloudflare Pages Functions.

| Endpoint | Auth | Description |
|----------|------|-------------|
| `GET /api/contributors` | Public | Returns deduplicated contributor list from KV (no emails exposed) |
| `POST /api/submit` | Access | Looks up caller's handle from KV; fires `repository_dispatch` to trigger ingest |
| `GET /api/profile` | Access | Returns caller's profile from KV |
| `POST /api/profile` | Access | Updates caller's name/url/bio in KV (handle and email are immutable) |
| `GET /api/admin/handles` | Admin | Lists all KV entries including emails |
| `POST /api/admin/handles` | Admin | Creates or updates a KV entry `{ email, handle, name, url, bio }` |
| `DELETE /api/admin/handles` | Admin | Removes a KV entry by email |
| `GET /api/notes-parquet` | Public | Range-aware R2 proxy — serves `notes.parquet` for DuckDB-WASM HTTP range reads |
| `POST /api/embed` | Public | Returns a Workers AI embedding vector for a query string |
| `POST /api/chat` | Public | Streams a Workers AI chat response grounded in Oracle context |

## Article submission pipeline (end to end)

1. Contributor visits `worldmachines.org/submit` → Access email OTP gate admits them.
2. Form submits to `POST /api/submit`.
3. Function reads `Cf-Access-Authenticated-User-Email`, looks up their handle in KV, fires `repository_dispatch` to GitHub with the submission payload.
4. **`ingest.yml`** runs: `scripts/ingest.py` extracts full text via `trafilatura`, writes `website/content/articles/<slug>.json`.
5. `scripts/build.py` regenerates `index.html`, `contributions.html`, `resources.html`.
6. Workflow commits changes and deploys to Pages via wrangler.

---

## Oracle pipeline (raw-notes → searchable corpus)

1. Contributors push Markdown notes to `raw-notes/<name>/`.
2. **`notes-ingest.yml`** triggers on any push touching `raw-notes/`: runs `tools/notes-pipeline/notes_to_parquet.py`, which parses all notes, generates embeddings via Workers AI, and writes `notes.parquet`.
3. The workflow uploads `notes.parquet` to the `worldmachines-notes` R2 bucket.
4. `oracle.html` loads DuckDB-WASM in the browser; on query, it fetches `notes.parquet` via HTTP range reads through `/api/notes-parquet`, runs a vector similarity search, and streams the result through `/api/chat`.

## Static build (`scripts/build.py`)

Generates four HTML files from source data:

| Output | Source |
|--------|--------|
| `index.html` | `blurbs.md` + article metadata |
| `contributions.html` | `content/articles/` (type: contribution) |
| `resources.html` | `content/articles/` (type: resource) |
| `devlog.html` | `devlog.md` |

Run locally to verify changes before deploying:

```bash
cd website
/opt/homebrew/bin/python3 scripts/build.py
# open index.html, contributions.html, resources.html, devlog.html to verify
```

Changes to `blurbs.md` or `devlog.md` trigger an automatic rebuild and deploy via `rebuild.yml`. All other changes to article content or build logic require a manual rebuild and deploy.

## GitHub Actions secrets

| Secret | Used by | Purpose |
|--------|---------|---------|
| `CLOUDFLARE_API_TOKEN` | `ingest.yml`, `rebuild.yml` | Pages deploy (Pages: Edit scope) |
| `CLOUDFLARE_ACCOUNT_ID` | `ingest.yml`, `rebuild.yml`, `notes-ingest.yml` | Cloudflare account identifier |
| `CF_AI_TOKEN` | `notes-ingest.yml` | Workers AI: Read (scoped narrowly) |
| `CF_R2_TOKEN` | `notes-ingest.yml` | R2 Storage: Edit (scoped narrowly) |

The AI and R2 tokens are intentionally scoped separately from the main Pages deploy token.
